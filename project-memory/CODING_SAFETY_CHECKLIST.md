# Coding Safety Checklist

Use this checklist **before, during, and after** every code change in this repository.  
Aligned with `.cursor/rules/project-memory.mdc` and `PROJECT_MEMORY.md`.

---

## Before coding

- [ ] **1. Read** `project-memory/PROJECT_MEMORY.md` first.
- [ ] **2. Load** the relevant summary from `project-memory/summaries/`:

  | If you are working on… | Load |
  |------------------------|------|
  | Features, data flow, performance, web/mobile structure | `architecture_summary.md` |
  | Deploy, env, Firebase indexes/rules, stores, DNS | `deployment_summary.md` |
  | Integrations, Firestore/Auth/Storage, serverless, third-party APIs | `api_summary.md` |
  | Bugs, regressions, incident-style fixes | `known_bugs.md` |

- [ ] **3. Verify current behavior in source** — read the files you will touch; trace call paths; confirm existing patterns in `src/services/`, `src/api/`, `mobile/` before editing. Do not assume root-level `*_FIX.md` guides match live code.
- [ ] **4. Plan minimal diffs** — smallest change that satisfies the request; no drive-by refactors, renames, or duplicate systems/APIs.

---

## While coding

- [ ] **5. Do not touch vendor or autogenerated files**

  - `node_modules/`
  - `dist/` (unless user explicitly requests a build-output change)
  - Expo/local tooling output under `mobile/.expo/`
  - Generated manifests, service workers, or lockfile-only dependency bumps without explicit request

- [ ] **6. Do not modify deployment configs** unless the user explicitly asks

  - `vercel.json`
  - `firestore.rules`, `firestore.indexes.json`
  - `mobile/app.json`, EAS profiles
  - `.env` / production secrets on Vercel or EAS

- [ ] **7. Preserve critical behavior**

  | Area | Rule |
  |------|------|
  | **Admin routing** | Do not weaken, bypass, or reroute admin pages/guards; respect `users.role` RBAC. |
  | **Session logic** | Preserve auth persistence, login/logout flows, and protected-route behavior (web + mobile). |
  | **Firebase / FCM** | Keep existing Auth, Firestore, Storage, and push/FCM integration paths; do not add parallel Firebase clients. |
  | **Agents SQL / display** | Keep data access (queries/SQL or service layer) separate from UI/display components; do not mix fetch logic into presentational components or duplicate query paths. |

- [ ] **Repo layout:** web at repo root (`src/`); mobile only in `mobile/`; sync listing constants via `npm run sync-listings` when web constants change.

- [ ] **Prefer** production-safe, incremental changes over large rewrites.

---

## After changes

- [ ] **8. Report** to the user (required):

  ### Files changed
  - List each path touched.

  ### Why changed
  - One line per file or logical group — tie to the user request.

  ### Risks
  - Regressions (auth, admin, listings, messaging, deploy, indexes, TWA/PWA).
  - Config or env assumptions.
  - Mobile/web drift if only one side was updated.

  ### How to test
  - Concrete steps: commands (`npm run dev`, `npm run build`, Expo start), pages/routes, and Firebase Console checks if relevant.
  - Reference `TESTING_FLOW.md` or `mobile/docs/IOS_ANDROID_RELEASE_CHECKLIST.md` when applicable.

---

## Quick “stop” signals

Do **not** proceed without explicit user approval if the change would:

- Add a second API client, service layer, or deploy path for something that already exists
- Edit `firestore.indexes.json` or rules without a stated Firebase task
- Change admin-only routes or role checks
- Merge agents/data layer with UI-only components
- Rely on stale docs (`zar-746103b7/`, `zarmongolia.com`, wrong Firebase project IDs) instead of verified code/console state
